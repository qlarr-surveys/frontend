Single-Question Inline Preview — How it executes

From the moment a user clicks the Preview (👁) action on a question, to the live question rendered in the design canvas — and the exact Qlarr engine functions that run in the browser.

React / Redux (UI + state) App service code Qlarr engine (Kotlin→JS) & vendored runtime

At a glance

The preview never talks to the server. It rebuilds the unsaved design into the engine's DSL, runs the same Qlarr engine the backend uses — compiled to JavaScript — entirely in the browser, trims the result down to the one question, and feeds it to the normal survey renderer inside an isolated Redux store.

1Click → editState

The 👁 button lives in the question's action toolbar. Its click is stopPropagation()'d (so it doesn't trigger canvas selection) and calls onPreview, which fires two Redux actions: one selects the question, one toggles the preview flag.

📄 src/components/design/ActionToolbar/index.jsx
const onPreview = useCallback(() => {
  // select this question so its setup follows it…
  dispatch(setup({ code, rules: setupOptions(type) }));
  // …and toggle the inline preview shown above the question card
  dispatch(togglePreviewQuestionCode(code));
}, [dispatch, code, type]);
📄 src/state/edit/editState.js — the only preview state in the design store
togglePreviewQuestionCode: (state, action) => {
  state.previewQuestionCode =
    state.previewQuestionCode === action.payload ? null : action.payload;
},

2GroupDesign mounts the preview

Every question in the design canvas is wrapped by a collapsible region. When previewQuestionCode matches that question's code, the <Collapse> expands and mounts the preview component (mountOnEnter / unmountOnExit means it's created fresh on open and destroyed on close — only one is ever alive).

📄 src/components/Group/GroupDesign.jsx
<Collapse in={previewQuestionCode === quest.code} mountOnEnter unmountOnExit>
  <InlineQuestionPreview
    code={quest.code}
    onClose={() => dispatch(setPreviewQuestionCode(null))} />
</Collapse>

On mount: an isolated run store

The design page runs on the manageStore, which has no runState. A live survey runs on a different store entirely. So the preview spins up its own private store for its lifetime — it never collides with a real run session and never triggers the design autosave.

📄 src/components/design/InlineQuestionPreview/index.jsx
function createPreviewStore() {
  return configureStore({ reducer: { templateState, runState } });
}
// created once per mount, kept in a ref for the component's whole life

3Debounced recompute — the heartbeat

One effect drives everything. It re-runs whenever the previewed question's node changes, the survey structure changes, or the language changes — but waits 400 ms so typing doesn't recompile on every keystroke. It reads the design store with store.getState() directly (not a dispatch) to stay off the autosave middleware.

📄 InlineQuestionPreview/index.jsx — inside the debounced setTimeout
const surveyJson = assembleSurveyJson(store.getState().designState);   // step 4
const computed   = await compileAndNavigate(surveyJson, lang, "offline"); // step 5–6
const pruned     = pruneToSingleQuestion(computed, code);              // step 7

previewStoreRef.current.dispatch(
  stateReceived({ response: pruned, preview: true, singleQuestion: true }) // step 8
);
Why "offline"? In online mode the engine compiles an offline-only question (barcode / photo / video capture) to relevance = false and hides it. Compiling in offline mode keeps every question type renderable in the preview.

4assembleSurveyJson — flat tree → engine DSL

Redux stores the design flat (each node keyed by its code, children referenced by id). The engine wants a nested DSL: Survey → groups → questions → answers. assembleSurveyJson walks the flat tree and rebuilds that nesting, merging each child's identity ref with its body.

📄 src/utils/design/utils.js
const CHILDREN_KEY_BY_LEVEL = ["groups", "questions", "answers"];

const assembleNode = (designState, ref, level) => {
  const node = designState[ref.qualifiedCode ?? ref.code];
  const { children, ...body } = node;
  const assembled = { ...ref, ...body };

  // strip mode_relevance: the engine DERIVES it internally but REJECTS it on input
  assembled.instructionList = assembled.instructionList
    .filter(i => i && !ENGINE_DERIVED_INSTRUCTION_CODES.has(i.code));

  // recurse: this level's children become groups / questions / answers
  assembled[childrenKeyForLevel(level)] = children.map(c => assembleNode(...));
  return assembled;
};
The mode_relevance gotcha. The backend persists a derived mode_relevance instruction into saved offline questions. The vendored engine generates that instruction itself but throws “Invalid JSON for instruction” if you feed it back in. Stripping it here is what let offline questions preview without errors.

5compileAndNavigate — the engine boundary

This is the heart. It mirrors, fully in-browser, what the backend does to turn a design into a runnable survey. Three things get injected into the page as <script> tags, then two engine use-cases run.

5a · Load the engine + runtime (once)

The engine bundle does not contain navigate or qlarrStateMachine. The Kotlin engine deliberately delegates all script evaluation (relevance, validation, skip logic — arbitrary JS) to a host-provided runtime. On the server that host is the backend; here those two vendored files are the host, which is what makes the preview backend-free.

5b · The two-stage engine run

📄 src/services/engine/qlarrEngine.js
const usecase = engine.com.qlarr.surveyengine.usecase;

// ── Stage 1: VALIDATE ───────────────────────────────────────────────
// parse the DSL → produce a runtime script + dependency/impact maps
validationJsonOutput = JSON.parse(
  usecase.ValidationUseCaseWrapper.create(surveyJson).validate()
);

// inject validationJsonOutput.script → defines window.qlarrRuntime
injectInlineScript(validationJsonOutput.script, RUNTIME_SCRIPT_ID);

// adapter the engine calls back into; we delegate to the vendored navigate()
const scriptEngine = { navigate: (s) => window.navigate(JSON.parse(s)) };

// ── Stage 2: NAVIGATE ───────────────────────────────────────────────
// compute the initial runtime state (values, relevance, validity…)
navigationJsonOutput = JSON.parse(
  usecase.NavigationUseCaseWrapper.init(
    "{}",                              // no saved responses
    JSON.stringify(validationJsonOutput),
    lang,
    exposed.NavigationMode.ALL_IN_ONE,
    null,
    exposed.NavigationDirection.Start,
    true,
    mode                               // SurveyMode.OFFLINE
  ).navigate(scriptEngine)             // ← engine calls scriptEngine.navigate(...)
);

return {
  survey:          navigationJsonOutput.survey,
  navigationIndex: navigationJsonOutput.navigationIndex,
  state: {
    qlarrVariables:  navigationJsonOutput.state.qlarrVariables,
    qlarrDependents: navigationJsonOutput.state.qlarrDependents,
  },
};
Each stage is wrapped in its own try/catch that re-throws as qlarr validate() failed: … / qlarr navigate() failed: … so a broken design surfaces as the preview's error box instead of a cryptic stack.

6Inside the engine: what validate() and navigate() actually do

ValidationUseCaseWrapper.validate() (Kotlin→JS, in the bundle)

Parses the DSL, type-checks instructions, and emits validationJsonOutput containing: the per-survey runtime script (becomes window.qlarrRuntime), the resolved survey tree, the impactMap (which component affects which), schema, and the dependency graph. It does not run any survey logic yet — it builds the machinery.

NavigationUseCaseWrapper.navigate(scriptEngine) (Kotlin→JS)

Decides which components are on the current page and asks the host to compute their starting values. The crucial move: it calls back into scriptEngine.navigate(script) — i.e. the vendored window.navigate() below — to actually evaluate the instructions.

window.navigate(input) (vendored — initial_script.js)

This is the JS runtime that produces the initial qlarrVariables map — one object per component holding its value, relevance, validity, etc.

📄 src/vendor/qlarr-engine/initial_script.js — the function the engine drives
function navigate(navigationInput) {
  qlarrVariables = {};
  // 1. make an empty object per component code
  codes.forEach(c => { eval(c + " = {};"); qlarrVariables[c] = eval(c); });

  // 2. seed values from saved data (type-checked)
  valuesKeys.forEach(k => qlarrVariables[a][b] = verifyValue(b, returnType, value));

  // 3. run each instruction (relevance / validity / skip / format …)
  sequence.forEach(si => {
    if (si.instruction.isActive)
      qlarrVariables[si.componentCode][code] =
        runInstruction(code, text, returnType);   // eval the JS expression
    else /* fall back to parsed text or defaultValue() */
  });
  return JSON.stringify(qlarrVariables);
}
HelperRole
runInstruction(code, text, type)evals the instruction's JS expression; if it throws or returns the wrong type, falls back to defaultValue().
verifyValue(code, type, value)Keeps a stored value only if it matches the declared returnType, else uses a default.
isCorrectReturnType(type, value)Type guard for boolean / date / int / double / list / enum / string / map / file.
defaultValue(code, type)Safe default. Note: relevance, conditional_relevance, validity default to true; value to undefined.

7pruneToSingleQuestion — focus on one question

The engine navigated the whole survey (it must, so cross-question logic resolves). This helper is the only place that knows the preview shows just one question: it returns a new response containing only the owning group with only the target question, and forces both their relevance on — so the shared renderer needs zero preview-specific branches.

📄 src/services/engine/qlarrEngine.js
export function pruneToSingleQuestion(response, qualifiedCode) {
  const ownerGroup     = groups.find(g => g.questions.some(q => q.qualifiedCode === qualifiedCode));
  const targetQuestion = ownerGroup?.questions.find(q => q.qualifiedCode === qualifiedCode);
  if (!ownerGroup || !targetQuestion) return response;   // safe pass-through

  const prunedGroup = { ...ownerGroup, inCurrentNavigation: true,
    questions: [{ ...targetQuestion, inCurrentNavigation: true }] };

  // full variables map kept (dependents still resolve); relevance forced true
  const variables = { ...response.state.qlarrVariables };
  variables[ownerGroup.code]  = { ...variables[ownerGroup.code],  relevance: true };
  variables[qualifiedCode]    = { ...variables[qualifiedCode],    relevance: true };

  return { ...response, survey: { ...response.survey, groups: [prunedGroup] },
           state: { ...response.state, qlarrVariables: variables } };
}
These four forced fields — group + question inCurrentNavigation on the tree, and group + question relevance in the variables — are the complete set the renderer keys off. That's why <Survey/>, <Group/>, and <Question/> stay pristine.

8stateReceived → render

The pruned response is dispatched into the isolated store. The reducer flattens it into the shape the renderer reads, and stashes the dependency graph in a module variable for the interaction loop.

📄 src/state/runState.js
stateReceived: (state, action) => {
  state.preview        = action.payload.preview;        // true → lenient error handling
  state.singleQuestion = action.payload.singleQuestion; // true → hide group header + nav
  qlarrDependents      = response.state.qlarrDependents;// module-scope, used on value change
  state.values         = response.state.qlarrVariables; // per-component runtime vars
  state.data.survey    = response.survey;
  state.data.navigationIndex = response.navigationIndex;
},

Then the component renders the real survey renderer inside the isolated store + a theme provider:

9Reactive value-change loop

The preview is fully interactive. When the respondent touches a control, the question dispatches valueChange; the reducer pushes that value through the dependency graph using the very same globals the engine produced — so relevance, validation and computed values update live, with no recompile.

📄 src/state/runState.js — setValueInState
try {
  window.qlarrStateMachine(           // ← from common_script.js
    state.values,                    // all component vars
    qlarrDependents,                 // who-affects-whom graph
    window.qlarrRuntime,             // per-survey script (from validate())
    componentCode, "value", value, "VALUE CHANGE"
  );
} catch (e) {
  if (!state.preview) throw e;       // real runs stay strict
  element["value"] = value;            // preview: record input, warn, don't crash
}
A debounced edit in the design (label, option, validation rule) instead invalidates the questionNode selector → the step 3 effect re-fires → the whole pipeline recompiles within ~400 ms. Two distinct loops: respondent input (instant, via qlarrStateMachine) vs. design edits (debounced full recompile).
